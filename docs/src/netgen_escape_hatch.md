# When and how to drop to `Delone.Internals`

Delone.jl's architecture has exactly two layers:

```
Delone                     exported Julian API (src/*.jl) — public, LLM-friendly
  └── Delone.Internals     strict 1:1 CxxWrap bindings from NetgenCxxWrap_jll
                            (raw Netgen/NGSolve backend; advanced/backend use)
```

`Delone.Internals` is generated by [CxxWrap](https://github.com/JuliaInterop/CxxWrap.jl)
from `NetgenCxxWrap_jll`'s C++ sources and keeps upstream Netgen/NGSolve names
**verbatim** — `GetNP`, `Refine`, `VolumeElement`, `Ngx_Mesh`, and so on. It is
loaded and initialized in `src/internals.jl` (`@initcxx` runs in `Internals`'s
own `__init__`) and is **not exported** from the top-level `Delone` module.
Every high-level Julian function in this package (`generate_mesh`, `refine!`,
`points`, …) is implemented by calling `Internals.*` internally and wrapping
the result in a Julian, 1-based, structured shape.

Most users and LLM agents should never need to write `Delone.Internals`
directly — that is the entire point of the Julian layer existing. This page
is for the minority of cases where you do.

## When to reach for it

Reach for `Delone.Internals` when you need a capability that upstream Netgen
genuinely exposes, but that has **no Julian wrapper yet**. Check
[`docs/API_COVERAGE.md`](https://github.com/ahojukka5/Delone.jl/blob/master/docs/API_COVERAGE.md)
in the package tree first — it lists every wrapped C++ class and method,
measured directly from the installed headers and the wrapper sources, so it
tells you definitively whether a capability is *reachable* even when no
Julian helper composes it yet.

A concrete example already documented in [Not yet wrapped](limitations.md):
`STLParameters` is fully wrapped in C++ (constructor, `yangle`, `contyangle`,
`edgecornerangle`, `chartangle`, `outerchartangle`, `usesearchtree`,
`recalc_h_opt`, all with getters and setters), but there is **no Julian
`STLOptions` wrapper** — the underlying `STLGeometry::GenerateMesh` override
copies a global C++ singleton instead of accepting a caller-supplied
`STLParameters` object, so a Julian composition layer cannot thread a value
through it yet. Until that C++-side gap is closed, constructing and reading
an `STLParameters` object is only possible via `Internals` directly:

```@example internals_escape_hatch
using Delone

p = Delone.Internals.STLParameters()
Delone.Internals.yangle(p)
```

```@example internals_escape_hatch
Delone.Internals.yangle!(p, 25.0)
Delone.Internals.yangle(p)
```

This is the general shape of a legitimate `Internals` use: the class and its
methods are confirmed wrapped (checked against `API_COVERAGE.md`), there is
no Julian entry point, and you accept that you are working with raw
CxxWrap-mediated C++ objects and 0-based conventions until this package (or
you, via a PR) closes the gap. [Not yet wrapped](limitations.md) keeps a
running list of exactly these Julian-layer gaps found so far
(`RestrictLocalH`/`SetLocalH` not influencing `generate_mesh`,
`FindOpenElements`/`FindOpenSegments` not reporting a count, `STLParameters`
above) so they are not rediscovered from scratch.

## The indexing seam: 1-based `Delone` vs. 0-based `Internals`

This is the single most common source of off-by-one bugs when mixing the two
layers, so it is worth stating plainly: **the Julian layer is always
1-based; `Delone.Internals` is always 0-based**, because it mirrors Netgen's
native C++ indexing exactly.

The clearest example is [`parent_nodes`](@ref) in `src/hierarchy.jl`, which
bridges the seam explicitly:

```julia
_ngx_to_1based(v::Integer) = Int32(v) + Int32(1)

function parent_nodes(m)
    nm = Internals.Ngx_Mesh(m)
    np = Internals.GetNP(m)
    P = Matrix{Int32}(undef, 2, np)
    buf = zeros(Cint, 2)
    for i in 1:np
        Internals.GetParentNodes(nm, i - 1, buf)          # 0-based query
        P[1, i] = _ngx_to_1based(buf[1])                  # back to 1-based
        P[2, i] = _ngx_to_1based(buf[2])
    end
    return P
end
```

Two conversions happen in that loop, in opposite directions:

- `i - 1`: the Julian loop variable `i` runs `1:np` (1-based), but
  `Internals.GetParentNodes` expects Netgen's native 0-based vertex index, so
  it is converted down by one **before** the call.
- `_ngx_to_1based(buf[k])`: Netgen's `GetParentNodes` writes back 0-based
  parent indices (with `-1`-as-"none" translating to Netgen's own `0`
  sentinel once shifted), so the result is converted back up by one **after**
  the call, giving the public `parent_nodes(mesh)` result its documented
  1-based convention with `0` meaning "none" — matching the rest of this
  package's public API (`tetrahedra`, `surface_triangles`, `cell_regions`, …
  are all 1-based).

The same pattern recurs in `parent_elements`/`parent_surface_elements`
(`src/hierarchy.jl`) and throughout `src/extraction.jl`/`src/tags.jl`: any
loop that calls an `Internals.*` accessor subtracts one from a Julian
1-based index going in, and any raw index Netgen hands back gets `+1`-ed
before it appears in a public Delone return value. If you write code that
calls `Internals.*` directly, you are responsible for doing this conversion
yourself — `Internals` will not warn you, since from its point of view
`0` is simply the first valid index, not an error.

## What this package does not do with `Internals`

Advanced/backend use of `Internals` is expected for genuinely missing
capabilities, but it is not a substitute for using the Julian layer's
existing coverage, and it is not meant to leak into public Delone.jl
return values: [`AGENTS.md`](https://github.com/ahojukka5/Delone.jl/blob/master/AGENTS.md)'s
convention is that reports are structured Julia types that never expose raw
`Internals` handles in a public field (`test/llm_feedback.jl` has an explicit
regression test for exactly this: `!(r.validation isa Delone.Internals.Mesh)`).
If you find yourself building a reusable composition on top of `Internals`
calls, consider whether it belongs in `src/` as a new Julian helper instead
of living only in application code — see `AGENTS.md`'s "Adding new bindings"
policy (new C++ bindings belong in `NetgenCxxWrap_jll`; Julian composition
belongs in `Delone.jl`).
